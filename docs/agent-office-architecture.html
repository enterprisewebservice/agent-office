Agent Office Architecture

A shared multi-agent control plane and visual workspace built on OpenClaw, OpenShift, GitOps, and Developer Hub. The key design point is that Agent Concierge can use OpenAI for top-level interaction while delegating operational work through the Claude Code subscription skill, which keeps coding and operator-style actions off the expensive API path.

Primary Namespace agent-office
Developer Hub Namespace rhdh-test
GitOps Namespace openshift-gitops
Custom Runtime Repo deanpeterson/openclaw-infra

Simple Meeting Summary

Agent Office is one shared application. Each new employee agent is created as its own GitOps-backed component, deployed into OpenShift, registered in Developer Hub, and then discovered by Agent Office so it appears both in the catalog and in the office UI.

  • Agent Concierge is the front desk agent that helps create and manage other agents.
  • Agent creation goes through the Agent Office backend, not by spinning up another copy of Agent Office.
  • Developer Hub tracks the agents as components.
  • Argo CD applies each agent’s repo into the cluster.
  • The visual office is a second UI layer on top of the same backend so agents feel like employees with desks and workstations.

Why the Claude skill matters: expensive Claude API usage is avoided for coding and operator-style tasks by using the Claude Code subscription skill. That keeps recurring costs much lower while still giving agents access to strong coding and CLI-driven execution.

1. System Context

The key separation is between the shared office application and the individual employee agents. Agent Office is one app. Agents are separate GitOps-backed components that Agent Office discovers and presents.

2. Agent Creation Flow

The important operational point is that Claude Code subscription is used for the actual create action, but the system of record is still the shared Agent Office backend plus GitOps, Developer Hub, and Argo CD.

3. Runtime Request Flow

Conversation Path

  • The browser talks to Agent Office over websocket.
  • Agent Office proxies the conversation to the selected OpenClaw agent gateway.
  • The selected agent decides whether to answer directly or call a skill/tool.
  • For concierge operations, the preferred path is OpenAI for top-level dialog plus Claude Code subscription for execution-heavy work.

What the UI Adds

  • Activity panel shows tool and skill usage per turn.
  • Session panel shows cached gateway state, OpenClaw chat count, and active versus historical Claude sessions.
  • Visual Office presents agents as employees in a navigable office instead of a plain list.
  • OpenAI TTS upgrades the spoken voice layer without changing tool or skill behavior.

4. Repos and Ownership

Core Repos

  • agent-office: shared web app, backend, visual office UI, chat/session/TTS behavior.
  • openclaw-infra: custom runtime and Claude bridge behavior.
  • onboarding-agent-agent-gitops: source of truth for Agent Concierge deployment and prompt/config.
  • tssc-dev-multi-ci: scaffolder template that produces new agent repos and manifests.

What Lives Where

  • agent-office namespace: shared app plus deployed agents.
  • rhdh-test namespace: Developer Hub and scaffolder UI.
  • openshift-gitops namespace: Argo CD applications syncing agent repos.
  • Quay image registry: custom Agent Office and OpenClaw images.

5. Design Rationale

Why This Model Works

  • One shared office app avoids duplicating the control plane for every employee agent.
  • Per-agent GitOps repos make lifecycle, deletion, and catalog registration predictable.
  • Developer Hub provides the familiar software catalog abstraction for agents.
  • The visual office makes the system easier to operate because people naturally think about long-running agents as employees with workstations.

Operational Risks to Watch

  • Do not let OpenClaw runtime images drift silently from the known-good custom build path.
  • Keep image tags versioned; avoid relying on mutable rollback points.
  • Keep session state visible and resettable because bad historical context can contaminate operational turns.
  • GitOps controller health matters directly; if Argo is unhealthy, agents can appear “created” in chat but not actually land in-cluster.
