Agent Office Architecture with Security and AgentOps Context

This version explains not just the current Agent Office implementation, but also where Red Hat security, operational controls, and the emerging AgentOps direction fit. It separates what is live today from what is future-facing or optional so the model is accurate in a meeting.

Short Version

Today, Agent Office is a practical operations layer on top of OpenClaw. Security is already in the mix through namespace isolation, GitOps-controlled deployment, secret management, runtime permission boundaries, and a custom OpenClaw bridge in openclaw-infra.

The Red Hat AgentOps direction, including Kagenti-style governance and operational control, is the logical next layer above this: policy, tracing, guardrails, lifecycle controls, and better standardized agent operations.

How to Say It in a Meeting

“We already have a working multi-agent platform on OpenClaw, OpenShift, GitOps, and Developer Hub. The custom `openclaw-infra` layer is where we shape runtime behavior and the Claude Code skill integration. The Red Hat security and AgentOps story sits around that as the governance and operational layer: policy, observability, permissioning, and controlled agent lifecycle.”

1. Security and Operations Stack

The most important distinction: openclaw-infra is part of the current runtime and security layer. Kagenti or broader AgentOps capabilities are a higher-level governance and operations layer around that runtime.

2. Where openclaw-infra Fits

What It Is

  • The custom OpenClaw infrastructure and image lineage used to shape runtime behavior.
  • The place where the Claude bridge extension lives.
  • The place where permission defaults, CLI invocation behavior, and tool/skill integration can be tuned.

Why It Matters

  • It is not just “build plumbing.”
  • It directly affects security boundaries, runtime compatibility, and how the Claude Code subscription skill behaves.
  • It is the layer that must stay stable if you want to avoid regressions in agent behavior.

How to Explain It

  • “openclaw-infra is our runtime control point.”
  • “It is where we encode the behavior that upstream OpenClaw does not give us out of the box.”
  • “That includes the Claude Code skill path and some of the practical security behavior.”

3. Security Boundaries in the Current System

Already Present Today

Namespaces Service Accounts RBAC Secrets GitOps drift control Session reset controls Skill/tool activity visibility
  • Agents run inside OpenShift and inherit cluster-level identity and policy controls.
  • Per-agent deployment goes through GitOps rather than uncontrolled manual mutation.
  • Secrets stay in-cluster and should not be committed into GitOps repos.
  • The Agent Office UI now exposes enough session state to detect stale or contaminated runtime state.

Why This Is Security-Relevant

  • Long-running agents are not just LLM prompts; they are software operators with memory, tools, and persistence.
  • That means permission scope, auditability, reproducibility, and rollback discipline matter as much as prompt quality.
  • OpenClaw plus GitOps makes the agent lifecycle tractable, but governance still benefits from a stronger AgentOps layer.

4. Where Kagenti Fits

The cleanest way to describe Kagenti is as an overlay or control layer around the agents you already have, not as a replacement for Agent Office or for the custom OpenClaw runtime work you already did.

Important accuracy note: in your current environment, Kagenti is best described as adjacent or forthcoming architecture rather than a core live dependency unless you have explicitly deployed it. The current live system is already doing useful security-sensitive work through OpenShift, GitOps, and the customized OpenClaw runtime.

5. Current vs. Future Talking Points

What Is Live Now

  • Shared Agent Office app and visual office UI.
  • GitOps-backed per-agent deployment and deletion.
  • Developer Hub component registration.
  • Custom `openclaw-infra` runtime behavior.
  • OpenAI for top-level interaction plus Claude Code subscription skill for real work.
  • Visible tool/skill/session state in the chat experience.

What Fits the Red Hat AgentOps Story

  • More formal policy and guardrails around agent tools and actions.
  • Stronger traceability and audit for agent decisions and executions.
  • Standardized lifecycle and governance for agent deployment, modification, and retirement.
  • A clearer security plane for enterprise operations teams.

6. Simple Meeting Script

“The live system today is already a real multi-agent operations platform on OpenShift. Agent Office is the shared control plane and visual workspace. Each agent is a GitOps-backed component that also shows up in Developer Hub.
